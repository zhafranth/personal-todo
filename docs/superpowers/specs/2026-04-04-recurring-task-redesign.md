# Recurring Task Redesign

## Summary

Redesign the recurring task system from an "auto-advance on completion" model to a "scheduler-generated instances" model. Recurring tasks are now defined in a separate `recurring_definitions` table, and a dedicated in-process scheduler generates task instances on the 1st of each month at 00:00 WIB (Asia/Jakarta). Completed tasks remain visible and are not modified or deleted. Additionally, add support for a `monthly_last_day` recurrence rule and a corresponding DatePicker quick-select chip.

## Current Behavior

When a recurring task is marked complete, the backend (in the same transaction) resets `is_completed = false` and advances `due_date` to the next occurrence. The task never truly stays "done" — it immediately reappears with a new due date.

## New Behavior

- Marking a recurring task as done simply sets `is_completed = true`. No side effects.
- Old completed tasks remain in the database and are visible in the UI.
- A dedicated scheduler generates the next task instance on the 1st of each month at 00:00 WIB.
- Recurring logic is driven by a new `recurring_definitions` table, not by the task row itself.

## Database

### New table: `recurring_definitions`

```sql
CREATE TABLE recurring_definitions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  section_id UUID NOT NULL REFERENCES sections(id) ON DELETE CASCADE,
  title TEXT NOT NULL,
  description TEXT,
  priority TEXT NOT NULL DEFAULT 'medium',
  recurrence_rule TEXT NOT NULL,
  next_due_date DATE NOT NULL,
  is_active BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

- `recurrence_rule`: `'daily'`, `'weekly'`, `'monthly'`, `'yearly'`, `'every_N_days'`, `'every_N_weeks'`, `'every_N_months'`, `'monthly_last_day'`
- `next_due_date`: the due date for the NEXT task to be generated by the scheduler
- `is_active`: allows pausing/resuming without deleting the definition

### Alter table: `tasks`

```sql
ALTER TABLE tasks
  ADD COLUMN recurring_definition_id UUID REFERENCES recurring_definitions(id) ON DELETE SET NULL;
```

- Nullable FK linking a generated task back to its definition
- `recurrence_rule` column on tasks remains as a read-only display label

## Backend: Recurrence Logic

### `recurrence/recurrence.go` updates

- `Validate()`: accept `"monthly_last_day"` as a valid rule
- `Next()`: for `monthly_last_day`, compute the last day of the next month
  - Example: 30 April -> 31 May, 31 May -> 30 June

### Remove auto-advance logic

Remove lines 147-161 in `repository/task.go` (`TaskRepo.Update`) that auto-reset `is_completed` and advance `due_date` when a recurring task is completed.

## Backend: Recurring Task Scheduler

A new goroutine `RecurringScheduler`, separate from the existing reminder scheduler.

### When it runs

- **On server start**: immediate catch-up check
- **Daily at 00:00 WIB (Asia/Jakarta)**: via timer that calculates duration until next midnight WIB

### Logic per tick

1. Query `recurring_definitions` WHERE `is_active = true` AND `next_due_date <= today` (handles all frequencies: daily tasks generate every day, weekly every week, monthly on the 1st of the month, etc. Also catches up missed runs.)
2. For each definition:
   - **Idempotency check**: if a task with matching `recurring_definition_id` + `due_date` already exists, skip
   - INSERT new task into the appropriate section with `due_date = next_due_date`, `recurrence_rule` copied as label, `recurring_definition_id` set
   - UPDATE `next_due_date` on the definition to the next period via `recurrence.Next()`
3. One transaction per definition

### Examples

**Monthly on the 25th:**
- User creates recurring April 25 -> task April created immediately, `next_due_date` = May 25
- May 1 00:00 WIB -> scheduler generates task May 25, sets `next_due_date` = June 25
- June 1 00:00 WIB -> generates task June 25, `next_due_date` = July 25

**Monthly last day:**
- User creates recurring last day of April (April 30) -> task created, `next_due_date` = May 31
- May 1 00:00 -> generates task May 31, `next_due_date` = June 30

**Startup catch-up:**
- If server was down on May 1 and starts on May 3, scheduler still generates because `next_due_date` (May 25) <= end of current month (May 31)

## API Endpoints

### New endpoints

- `POST /api/recurring-definitions` — Create definition + first task
  - Body: `{ section_id, title, description?, priority, recurrence_rule, due_date }`
  - Response: `{ definition, task }`

- `GET /api/recurring-definitions` — List all definitions for the authenticated user

- `PATCH /api/recurring-definitions/:id` — Update definition (title, section, rule, is_active, etc.)

- `DELETE /api/recurring-definitions/:id` — Delete definition (already-generated tasks remain)

### Modified endpoints

- `PATCH /api/tasks/:id` — Remove auto-advance recurrence logic. `recurrence_rule` becomes read-only on tasks.

## Frontend

### RecurrencePicker

- Add new preset chip: "Last day of month" with value `"monthly_last_day"`
- Update `formatRecurrenceRule()` to return `"Last day of month"` for this rule

### DatePicker

- Inside the calendar popover, add a chip area below the calendar grid (before "Clear date")
- Chip: "Last day of month" — sets value to the last day of the currently viewed month
- Only visible when recurrence rule is `monthly_last_day` (passed as a prop)

### CreateTaskPage

- When user selects a recurrence rule, form submits to `POST /api/recurring-definitions` instead of `POST /api/tasks`
- First task is created server-side as part of the definition creation

### TaskItem & DayTaskCard

- Continue displaying recurrence badge from `task.recurrence_rule` (read-only)
- Mark done simply toggles `is_completed`, no special behavior

## Timezone

All scheduler timing uses `Asia/Jakarta` (WIB, UTC+7). This is hardcoded, not per-user.

## Migration Strategy

Existing tasks with `recurrence_rule` set need to be migrated:
1. For each task with a non-null `recurrence_rule` and `is_completed = false`, create a corresponding `recurring_definitions` row
2. Set `next_due_date` based on the task's current `due_date` + one period
3. Link the task to the new definition via `recurring_definition_id`
4. Tasks that are already completed with a `recurrence_rule` can be left as-is (orphaned label)
